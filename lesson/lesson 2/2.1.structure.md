# โครงสร้าง Directory ของ Nuxt 4 (Nuxt 4 Directory Structure)

Nuxt 4 มีการจัดโครงสร้างโปรเจกต์ที่เป็นระเบียบ โดยใช้รูปแบบที่เรียกว่า **Shared `app/` Directory** ซึ่งเป็นการแยกส่วนโค้ดฝั่งหน้าบ้าน (Client-side/UI) ออกจากโค้ดฝั่งหลังบ้าน (Server-side) อย่างชัดเจน ช่วยให้การพัฒนาและการทำ Hot Module Replacement (HMR) รวดเร็วมากยิ่งขึ้น

# การแบ่งเลเยอร์การทำงานใน Nuxt 4

1. Client-side (หน้าบ้าน)

- การทำงาน: เว็บเบราว์เซอร์ของผู้ใช้ (Chrome, Safari, Edge)
- คืออะไร: คือส่วนที่ผู้ใช้มองเห็นและโต้ตอบด้วยได้ทันที (User Interface)
- หน้าที่หลัก:
  - Interactive: จัดการการคลิกปุ่ม, การพิมพ์ฟอร์ม, การทำ Animation
  - State Management: เก็บข้อมูลชั่วคราวขณะใช้งานเว็บ (เช่น สินค้าในตะกร้า)
  - Browser API: เข้าถึงความสามารถของเครื่องผู้ใช้ เช่น กล้อง, ตำแหน่ง (GPS) หรือ LocalStorage
- ข้อจำกัด: ไม่สามารถเชื่อมต่อฐานข้อมูลตรงๆ ได้ เพราะไม่ปลอดภัย (User เห็นโค้ดได้หมด)

2. Server-side (หลังบ้าน)

- การทำงาน: Bun Runtime (เครื่องแม่ข่าย)
- คืออะไร: คือส่วนประมวลผลที่อยู่เบื้องหลัง จัดการโดย Nitro (ใน Nuxt) และ Elysia.js (ใน API Layer ของคุณ)
- หน้าที่หลัก:
  - Data Access: เชื่อมต่อ Database ผ่าน Drizzle ORM เพื่อดึงหรือบันทึกข้อมูล
  - Security: ตรวจสอบสิทธิ์ (Authentication), เก็บความลับ (API Keys), และจัดการ Business Logic สำคัญๆ
  - SSR (Server-Side Rendering): ใน Nuxt 4 ฝั่ง Server จะทำหน้าที่ "ปั้นหน้าเว็บ" ให้เสร็จก่อนส่งไปให้เบราว์เซอร์ เพื่อให้เปิดปุ๊บเห็นเนื้อหาปั๊บ (ดีต่อ SEO)

# โครงสร้างมาตรฐานของ Nuxt 4:

```text
nuxt-app/
├── app/                 # โฟลเดอร์ศูนย์กลางสำหรับเก็บโค้ด Vue.js (หน้าบ้าน)
│   ├── assets/          # สิ่งที่ต้องประมวลผล (CSS, SCSS, รูปภาพ)
│   ├── components/      # UI วิดเจ็ตย่อยสำหรับนำไปใช้ซ้ำ
│   ├── composables/     # ฟังก์ชัน Logic ที่ใช้ซ้ำได้ (Auto-imported)
│   ├── layouts/         # โครงสร้างหน้าเว็บหลักของโปรเจกต์
│   ├── middleware/      # ตัวดักจับและตรวจสอบก่อนเปลี่ยนหน้า
│   ├── pages/           # ส่วนสร้างหน้าเว็บตาม URL (Routing)
│   ├── plugins/         # โค้ดที่เรารันตอนแอปเริ่มต้นทำงาน
│   ├── utils/           # ฟังก์ชัน Helper หรือฟังก์ชันช่วยคำนวณ
│   ├── app.config.ts    # ไฟล์ตั้งค่าที่ฝั่งหน้าบ้านอ่านได้
│   ├── app.vue          # Component หลักของแอป
│   └── error.vue        # หน้าต่างแสดงความผิดพลาดของระบบ
├── public/              # ไฟล์ที่ฝั่งเบราว์เซอร์เข้าถึงได้ตรงๆ ไม่ผ่านการแปลง
├── server/              # หลังบ้าน (API Routes & Server Logic)
│   ├── api/             # ไฟล์สำหรับเขียน API Endpoint (/api/*)
│   ├── routes/          # ไฟล์สำหรับเขียน Server routes (/*)
│   └── middleware/      # ตัวดักจับ request ทุกครั้งที่ยิงเข้ามา
├── .nuxt/               # ระบบสร้างไฟล์ประมวลผลอัตโนมัติ (ห้ามแก้)
├── nuxt.config.ts       # ไฟล์คอนฟิกการตั้งค่าหลักของแอป
├── package.json         # รายชื่อเครื่องมือเสริม (Dependencies)
└── tsconfig.json        # ตั้งค่า TypeScript
```

## 📂 อธิบายเจาะลึก แต่ละโฟลเดอร์คืออะไร และทำอะไรใน Nuxt 4 ?

### 1. `app/assets/`

- **คืออะไร:** โฟลเดอร์สำหรับเก็บทรัพยากร (Assets) ที่ **"ต้องผ่านการประมวลผล"** ก่อนนำไปใช้งานจริงบนหน้าเว็บ
- **ใช้ทำอะไร:** ใช้เก็บไฟล์สไตล์หลักระดับ Global เช่นหน้าตั้งค่า `main.scss`, โค้ดต้นแบบของ Tailwind CSS หรือไฟล์รูปภาพ/ไอคอนที่อาจจะต้องทำการบีบอัดหรือแปลงนามสกุลก่อนใช้งาน โดยฝั่ง Nuxt จะใช้เครื่องมืออย่าง Vite เข้ามาจัดการคอมไพล์ส่วนนี้เป็นชั้นให้เอง

### 2. `app/components/`

- **คืออะไร:** พื้นที่สำหรับสร้างและเก็บชิ้นส่วน UI ย่อยๆ สำหรับประกอบร่างเป็นหน้าเว็บ (เช่น ส่วนปุ่มกด Button, การ์ดแสดงบทความ, หรือกล่องค้นหา) เพื่อให้นำกลับมาทำซ้ำใช้งานที่อื่นได้ (Reusable)
- **ทำงานอย่างไรใน Nuxt 4:** Nuxt จะจำลองและอ่านไฟล์ทั้งหมดในโฟลเดอร์นี้เพื่อทำระบบ **Auto-import** ให้เราแบบอัตโนมัติ ดังนั้นหากคุณสร้างไฟล์ `app/components/Card.vue` คุณจะสามารถพิมพ์แท็ก `<Card />` เรียกใช้การ์ดนั้นซ้อนในหน้าออฟฟิศอื่นได้เลยโดยไม่ต้องเขียนคำสั่ง `import Card from ...` ให้ยืดยาว

### 3. `app/composables/`

- **คืออะไร:** โฟลเดอร์เก็บ Logic และข้อมูลสถานะ (State) ที่ต้องการใช้งานร่วมกันในหลายๆ คอมโพเนนต์ โดยจะเขียนด้วยหลักการของ Vue Composition API (ชื่อฟังก์ชันส่วนใหญ่มักขึ้นต้นด้วย `use...` เช่น `useCart()`)
- **ใช้งานตอนไหน:** เวลาที่คุณทำการเชื่อมต่อฐานข้อมูล หรือระบบตะกร้าสินค้าข้ามหน้า ที่ค่าตัวแปรข้อมูลมันต้องอัปเดตติดต่อกันไปมาระหว่างหน้าเว็บ ไฟล์ระบบทั้งหมดในนี้จะถูกนำไปทำ **Auto-import** โดยอัตโนมัติ

### 4. `app/layouts/`

- **คืออะไร:** โฟลเดอร์ที่ใช้คอยกำหนด **"โครงร่างเทมเพลต"** หลักของหน้าเว็บไซต์
- **ทำงานอย่างไร:** ใช้สำหรับส่วนประกอบของหน้าที่จะไม่เปลี่ยนแปลงบ่อยๆ ในทุกหน้าจอ (เช่น การวาง Navbar เมนูไว้ด้านบนระดับสูงสุด และหน้า Footer ค้างไว้ตรงจุดล่างสุดตลอดเวลาเสมอ) โดยมันจะทำตัวเป็นเปลือกและรอรับเนื้อหาจากแฟ้ม `pages/` (ถูกแทรกเข้าไปผ่านแท็ก `<slot />`) ช่วยเพิ่มความเป็นระเบียบและลดการเขียนโค้ดซ้ำๆ

### 5. `app/middleware/`

- **คืออะไร:** ตัวกลางดักจับเหตุการณ์ Route ควบคุม **"ก่อน"** ที่จะอนุญาตให้ทำการสลับลิงก์เปลี่ยนหน้าทางฝั่งเบราว์เซอร์
- **ใช้งานตอนไหน:** ส่วนใหญ่จะใช้ดักตรวจสอบเส้นทาง เช่น สมมติเราสร้างหน้า Admin ไว้ หากผู้ใช้เปิดหน้านั้น แล้วดักเจอว่า **ผู้ใช้ยังไม่ได้ล็อกอินเข้าสู่ระบบ** Middleware ก็จะจัดการเด้งเตะผู้ใช้ออกไปป้อนรหัสที่หน้าลิงก์เข้าสู่ระบบอีกที

### 6. `app/pages/`

- **คืออะไร:** พื้นที่และโฟลเดอร์พระเอกสำหรับสร้างหน้าเพจ "หน้าเว็บ" (Pages)
- **ทำงานอย่างไรใน Nuxt 4:** หัวใจหลักของตัวจัดการหน้านี้เรียกว่า **File-based Routing** เพียงแค่คุณสร้างไฟล์นามสกุล `.vue` เพิ่มเข้าไป ระบบจะแปลงเป็นเส้นทางให้เข้าชมใช้งานในเบราว์เซอร์แบบ Auto ทันที ตัวอย่างเช่น:
  - สร้างไฟล์ `app/pages/index.vue` -> ทำเว็บหน้าหลัก และเข้าถึงเว็บผ่าน `http://yourweb/`
  - สร้างไฟล์ `app/pages/about.vue` -> เข้าถึงเว็บผ่าน `http://yourweb/about`

### 7. `app/plugins/`

- **คืออะไร:** พื้นที่เก็บโค้ดปลั๊กอินหรือไลบรารีของ Vue ที่เราต้องการลงทะเบียนและ **"ตั้งเปิดรอเพื่อสั่งให้มันรันทำงานตั้งแต่ก่อนเริ่มเข้าสู่แอปพลิเคชัน"** (เช่น ระบบตัวจัดการ Google Analytics, ฟอนต์ไอคอน หรือ 3rd Party Library ต่างเบราว์เซอร์)

### 8. `app/utils/`

- **คืออะไร:** พื้นที่เก็บฟังก์ชันช่วยเหลือ (Helper Functions) จิปาถะทั่วไปขนาดเล็ก
- **ทำงานอย่างไรใน Nuxt 4:** ใช้เก็บฟังก์ชันเพียวๆ ที่ไม่ได้เกี่ยวข้องกับหน้าเว็บ UI โดยตรง (เช่น ฟังก์ชันฟอร์แมตแปลงวันที่ให้อ่านง่าย หรือคำนวณราคาหักภาษีในตะกร้า) ฟังก์ชันทั้งหมดจะถูกนำไปตั้งเป็น **Auto-import** ให้อัตโนมัติ (คล้ายคลึง `composables/`)

### 9. `public/`

- **คืออะไร:** พื้นที่เก็บสื่อเก็บ **"ไฟล์สถิตย์"** (Static files) ดั้งเดิมที่ไม่ต้องการให้ผู้สร้างตัวแพ็กบิลด์มาแตะต้องดัดแปลงหรือแปลงร่างใดๆ ทั้งสิ้น ตัวอย่างเช่น พวกไฟล์ `robots.txt`, รูปภาพดิบที่จะไม่ถูกบีบอัด, ฟอนต์ หรือ Favicon (icon เว็บขนาดเล็ก)
- **ทำงานอย่างไร:** เบราว์เซอร์สามารถเรียกดาวน์โหลด/เข้าถึงไฟล์ดังกล่าวได้โดยตรงจากโฟลเดอร์ Root URL นอกสุดทันทีเมื่อระบุลิงก์ตรงๆ (ไม่ถูกย้ายเข้าไปตั้งใน `app/`)

### 10. `server/`

- **คืออะไร/ทำอะไรใน Nuxt 4:** จุดเด่นความเป็นอาวุธระดับโปร Full-stack ที่อนุญาตให้โปรเจกต์ของคุณสามารถจัดการฐานข้อมูลตั้งรันเป็น **Backend API** คู่กันเบื้องหลัง (Nitro Engine) ได้อย่างราบรื่น (จะแยกไฟล์กันไม่เกี่ยวข้องกันทางโครงสร้างเวลาแก้ไข UI ฝั่งหน้าบ้าน `app/`)
- **ตัวอย่าง:** หากคุณสร้างไฟล์โค้ด Backend อย่าง `server/api/hello.ts` คุณก็จะสามารถเอาไปป้อนยิงขอข้อมูลดึงข้อมูลมาแสดงผลด้วย (Fetch HTTP Request) โดยต่อผ่านลิงก์ `http://yourweb/api/hello` ได้แบบทันทีทันใด

### 11. `nuxt.config.ts`

- **คืออะไร:** ไฟล์แห่งจิตวิญญาณ จุดศูนย์กลางตั้งค่าการคอนฟิกทำงานทั้งหมดระดับ Global ของโปรเจกต์ Nuxt ของเรา ไม่ว่าจะเป็นการเปิดใช้งาน Modules ส่วนเสริม, จัดการโหลดข้อมูลโครงสร้าง SEO/Meta tag เบื้องต้น, จนถึงกำหนดตัวแปรความปลอดภัยรหัสผ่าน (Environment Variables) โฟลเดอร์จะถูกดันไว้นอกสุดเพื่อเป็นแม่แบบเสมอ
